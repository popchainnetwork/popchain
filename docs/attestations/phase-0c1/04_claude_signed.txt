  SECTION 1 — SCOPE OF MY CONTRIBUTION
═══════════════════════════════════════════════════════════════════════════

I participated as adversarial reviewer in six iterations across the
Phase 0b → Phase 0c.1 sequence. Specific contributions in this scope:

Phase 0b (pre-0c.1, contextual):
  - Confirmed RFC 8391 §4.1.7 reading: XMSS uses single shared pub_seed
    across all 2^h leaves; per-leaf separation via sk_seed and ADRS.
  - Confirmed proposed fix (new generate_keypair_with_seeds public fn,
    leaf_keypair propagating xmss_pub_seed) is correct and that simpler
    alternatives are cryptographically infeasible.
  - Reviewed KAT-10 candidate root 2935fde2…903606 from alpha and
    recommended strict 3-machine hex+pub_seed convergence gate before
    locking Anchor 3, with Anchor 2 / KAT-1 byte-equality regression
    guard as paranoia-tier defense.

Phase 0c.1 KAT-21..26 review (game 3 → game 5):
  - Reviewed v1 → v2 → v3 drafts of the persistence_stress test suite.
  - Identified four substantive bugs in v2 that needed to be fixed
    before commit:
      (1) KAT-22 synthetic crash state did not exercise the recovery
          contract it claimed to exercise (wrote post-flush state, not
          pre-flush state).
      (2) KAT-22 used "recovered >= prev_reserved_until" inequality where
          PARANOID semantics requires exact equality.
      (3) KAT-21 expected_after_recovery formula needed pinning against
          the actual clean-shutdown vs reopen-burn semantics in source.
      (4) KAT-26 phase 1/2 loop termination conditions used non-
          deterministic exit on current_state().idx instead of pinning
          last_signed.
  - Recommended replacing mem::forget(store) with direct on-disk
    overwrite for crash simulation (cleaner, no fd/lock leakage).
  - Recommended HashSet uniqueness check in KAT-22 inner loop.
  - Recommended addition of KAT-26 (backup-restore detection) as
    [#[ignore]] with explicit P1.6 epoch-counter TODO, on the grounds
    that it is the primary NIST XMSS deployment footgun and the test
    suite was incomplete without it as audit-trail.
  - Recommended addition of kat_state_serde_roundtrip as cross-cutting
    protection against silent endianness/padding regressions.
  - Flagged the v2 KAT-23 ambiguity (tests required Phase 0c.1.a guards
    that did not yet exist) and recommended landing guards + tests in
    one atomic commit.

PoCC standard draft brief (game 5):
  - Provided recommendations and reasoning for Q1 (block schema:
    inner-enum proof_payload), Q2 (2f+1 BFT with explicit N=3 caveat),
    Q3 (local-clock detection + median-of-cosigners canonical
    timestamp), Q4 (silence reward layered defense including mempool
    gossip evidence), Q5 (fork over migration), Q6 (structure-vs-
    parameters separation), Q7 (Anchor 1 in every block header),
    Q8 (stay with WOTS+/XMSS, reject BLS as non-PQ, flag XMSS h=10
    capacity question against alpha throughput as high-priority
    empirical unknown), Q9 (two-tier slashing: stake-slash for
    equivocation, jail-only for downtime), Q10 (PoW/PoS/PoH/PoSpace/PoP
    comparison table).
  - Flagged "Proof of Process" prior-art collision with BPM literature.
  - Flagged three structural omissions in the brief: partition
    liveness/safety choice, validator set membership transitions,
    light client / external verification design.

Phase 0c.1 commit verification (game 6):
  - Reviewed cross-arch verification output for the three machines.
  - Recommended commit + push immediately rather than batching.
  - Recommended roadmap swap: P1.7 (XMSS^MT) before P1.5/P1.6, on the
    grounds that hardening h=10 state machine that will be replaced
    by multi-tree variant before beta is largely wasted work.
  - Recommended P5 (Zymkey HSM) parallel to P1.7 due to hardware risk
    that should be discovered early, not late.
  - Recommended (d) on-chain epoch counter as primary mitigation for
    KAT-26, with (a) audit log as defense-in-depth and (c) HSM EEPROM
    deferred to Phase 1.
  - Refined the commit message to include explicit anchor-equality
    statement for self-documenting git log.
  - Flagged that the synthesized PoCC standard "consensus" items were
    not yet adversarially reviewed and should not be locked.
  - Flagged that the "AI-human cooperative authorship" framing for
    NLnet may be strategically counterproductive and recommended
    repositioning AI involvement as adversarial review methodology
    rather than co-authorship.
