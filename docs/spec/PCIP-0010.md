# PCIP-0010
## POP-2-COMPUTE Active Compute Profile

**Document ID:** PCIP-0010  
**Title:** POP-2-COMPUTE Active Compute Profile  
**Version:** 0.1-draft  
**Status:** Draft  
**Language:** English  
**Intended category:** Normative Profile Specification  
**Depends on:** PCIP-0001 (POP-1), PCIP-0002 (POP-2)  
**Date:** 2026-05-10

---

# 1. Scope

This document defines **POP-2-COMPUTE**, a normative scoring and derivation
profile for **POP-2** (PCIP-0002), specialized for active computational
workloads.

POP-2-COMPUTE extends POP-2 derivation rules to admit proofs of work
performed by computational devices (CPU, GPU, dedicated accelerators)
where the captured payload is not file content but the deterministic
output of a declared computational workload.

POP-2-COMPUTE defines:

- the canonical workload declaration object,
- required workload fields,
- compute backend abstraction,
- deterministic verification rules,
- compute scoring requirements,
- compute MBW derivation rules,
- cross-platform reproducibility requirements,
- validator recomputation requirements.

POP-2-COMPUTE does **not** define:

- specific compute kernel selection,
- hardware identity attestation (covered by future PCIP),
- tokenomics rates,
- consensus admission policy,
- transport protocols.

This document is a **profile**, not a new POP layer. The canonical
object remains POP-2 as defined in PCIP-0002. POP-2-COMPUTE specifies
how POP-1 captures of compute workloads MUST be structured and how
POP-2 derivation MUST proceed for such captures.

---

# 2. Conformance language

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**,
**SHALL NOT**, **SHOULD**, **SHOULD NOT**, **MAY** are to be
interpreted as normative requirements (RFC 2119 / RFC 8174).

---

# 3. Purpose

POP-2-COMPUTE exists to extend PCIP to active compute proofs while
preserving:

1. POP-1 canonical container semantics,
2. POP-2 deterministic proof semantics,
3. validator recomputation requirements,
4. ledger neutrality,
5. transport neutrality,
6. cross-platform reproducibility.

POP-2-COMPUTE enables:

1. industrial validators (T1-T4) submitting compute work,
2. AI validator class (T-AI) submitting verifiable inference,
3. cross-architecture compute proof admission,
4. deterministic MBW accounting for computational labor.

---

# 4. Design principles

POP-2-COMPUTE SHALL follow the principles below.

## 4.1 Canonical workload declaration

A compute proof MUST declare its workload deterministically. The
declaration MUST be sufficient for any conformant verifier to
reconstruct the workload and recompute its expected output.

## 4.2 Backend independence

The same workload MUST produce identical canonical output regardless
of compute backend (CPU, CUDA, ROCm, Metal, custom). Backend identity
is recorded but does not affect canonical proof identity.

## 4.3 Deterministic execution

All fields used in POP-1 / POP-2 derivation MUST be deterministic
functions of the workload declaration.

## 4.4 Recomputability

A verifier MUST be able to:

- accept the workload declaration,
- execute the workload on any conformant backend,
- obtain bit-identical output bytes,
- derive identical POP-1 metrics,
- derive identical POP-2 score and MBW.

## 4.5 No payload embedding

POP-1 binds to compute output **bytes**, not to embedded source data.
The workload declaration MUST be sufficient to regenerate output.

## 4.6 Profile neutrality

POP-2-COMPUTE is one of multiple POP-2 profiles. Other profiles
(file integrity, telemetry capture, AI inference) MAY coexist.
Profile selection is captured at deployment level.

# 5. Compute workload declaration

## 5.1 Canonical type

```text
ComputeWorkloadV1
 ├─ kernel_id: string (canonical kernel identifier)
 ├─ kernel_version: u32
 ├─ input_seed: Hash32 (32-byte deterministic seed)
 ├─ n_iterations: u64 (workload size parameter)
 ├─ output_size_bytes: u64 (declared output byte length)
 └─ params_hash: Hash32 (hash of additional kernel parameters)
```

## 5.2 Fields

### kernel_id

Type: string. REQUIRED.

Canonical identifier for the compute kernel. MUST be registered
in a kernel registry (out of scope for this document).

Examples (informative): `"sha256-chain-v1"`, `"blake3-merkle-v1"`,
`"matmul-fp32-v1"`.

### kernel_version

Type: u32. REQUIRED.

Version of the kernel implementation. MUST match across compute
backends. Different versions MUST produce different `kernel_id`
or different `kernel_version`.

### input_seed

Type: Hash32 (32 bytes). REQUIRED.

Deterministic seed for the workload. MUST be reproducible by all
verifiers.

### n_iterations

Type: u64. REQUIRED.

Number of iterations or work units. MUST scale linearly with MBW.

### output_size_bytes

Type: u64. REQUIRED.

Declared size of compute output in bytes. The actual output bytes
MUST equal this size exactly.

### params_hash

Type: Hash32. REQUIRED.

Hash of any additional kernel-specific parameters (block size,
matrix dimensions, etc.). For kernels with no extra parameters,
MUST equal the zero hash.

---

# 6. Relationship to POP-1

A POP-2-COMPUTE proof binds POP-1 to compute workload via:

1. POP-1 raw payload = compute output bytes (`output_size_bytes`),
2. POP-1 `sha256` and `blake3` MUST be computed over compute output bytes,
3. POP-1 metrics (`entropy_q32`, `compression_ppm`, `mad_q32`) MUST be
   computed over compute output bytes per PCIP-0001 §8,
4. POP-1 `machine_id` MUST identify the compute device or device cluster,
5. POP-1 `prev_capture_hash` MUST link to the previous accepted capture
   per PCIP-0001 §6.

The compute workload declaration is **not** part of the POP-1 container.
It is part of the POP-2-COMPUTE derivation context.

---

# 7. Backend abstraction

## 7.1 Backend types

A conformant POP-2-COMPUTE implementation SHALL support compute
backend identification:

```text
ComputeBackend
 ├─ Cpu
 ├─ Cuda
 ├─ Rocm
 ├─ Metal
 └─ Custom(string)
```

## 7.2 Backend independence requirement

For the same `ComputeWorkloadV1`, all conformant backends MUST produce
identical output bytes. Bit-level reproducibility is REQUIRED.

A backend that cannot guarantee bit-level reproducibility MUST NOT be
used for POP-2-COMPUTE proofs.

## 7.3 Backend identification

Backend identity MAY be recorded outside POP-2 canonical proof object
(e.g., in deployment-side metadata). Backend identity MUST NOT affect
POP-2 canonical proof hash.

---

# 8. Deterministic execution requirements

A conformant POP-2-COMPUTE workload kernel MUST satisfy:

## 8.1 Functional purity

For inputs (`input_seed`, `n_iterations`, `params_hash`), the kernel
MUST produce a single canonical output byte sequence. No environmental
input MAY influence output.

## 8.2 Floating point determinism

Kernels using floating-point operations MUST specify:

- IEEE 754 mode requirements,
- rounding mode,
- order of operations,
- reduction strategy.

Kernels that cannot guarantee bit-identical output across backends due
to floating-point non-associativity MUST either:

- specify a canonical reduction order, or
- restrict to integer-only computation.

Implementations using non-deterministic FP fast-math optimizations
(e.g., FMA contraction without explicit control) MUST NOT be used for
POP-2-COMPUTE.

## 8.3 Memory access patterns

Kernel output MUST NOT depend on memory layout, alignment, or
allocation pattern that varies across backends.

## 8.4 Concurrency determinism

Kernels using parallelism MUST guarantee output independence from
thread scheduling, work-group sizes, or warp configuration.

---

# 9. Verification requirements

A conformant POP-2-COMPUTE validator MUST be able to:

## 9.1 Workload reconstruction

Given `ComputeWorkloadV1` declaration, validator MUST reconstruct the
canonical compute task.

## 9.2 Output recomputation

Validator MUST execute the workload on a conformant backend and obtain
bit-identical output bytes.

## 9.3 POP-1 metric recomputation

Validator MUST recompute POP-1 fields per PCIP-0001 §11.

## 9.4 POP-2 derivation

Validator MUST derive POP-2 fields per PCIP-0002 §11.

## 9.5 Sampling-based verification (informative)

For very large workloads (high `n_iterations`), validators MAY accept
sampling-based verification:

- compute kernel produces checkpoint outputs at declared intervals,
- validator verifies a subset of checkpoints,
- subset selection MUST be deterministic from POP-1 `container_hash`.

This sampling rule is informative; conformant deployments MAY enforce
full recomputation.

---

# 10. POP-2-COMPUTE scoring profile

## 10.1 Scoring inputs

POP-2-COMPUTE `score_u64` MUST be computed deterministically from:

- POP-1 metric values (`entropy_q32`, `compression_ppm`, `mad_q32`),
- workload size (`n_iterations` from declaration),
- kernel-specific work coefficient (registered with `kernel_id`).

## 10.2 Scoring formula

```text
work_units = n_iterations * kernel_work_coefficient
metric_score = score(entropy_q32, compression_ppm, mad_q32, ScoringParamsV0)
score_u64 = saturating_mul(metric_score, work_units)
```

The `score` function follows PCIP-0002 §9.3 (deterministic,
reproducible, no hidden inputs).

`saturating_mul` MUST clamp to u64::MAX without wrap-around.

## 10.3 Kernel work coefficient

Each registered `kernel_id` MUST declare:

- a deterministic work coefficient,
- a reference benchmark hash rate or FLOP count,
- a normalized MBW factor.

Kernel registry rules are out of scope for this document.

---

# 11. POP-2-COMPUTE MBW derivation

`mbw_u64` MUST be derived from `score_u64` per PCIP-0002 §9.4.

For POP-2-COMPUTE, the canonical MBW transform is:

```text
mbw_u64 = score_u64 / 1000
```

with saturation at zero (no wrap-around).

This transform is a profile constant. Other profiles MAY use other
transforms.

---

# 12. Replay and continuity

POP-2-COMPUTE inherits replay semantics from POP-1 / POP-2 per
PCIP-0001 §12 and PCIP-0002 §12.

Each compute device MUST maintain its own capture chain. Cross-device
chain mixing is FORBIDDEN.

For a given `machine_id`:

- first compute proof MUST use zero `prev_capture_hash`,
- subsequent proofs MUST reference the previous accepted capture hash,
- replay enforcement MUST be active for compute proof admission.

---

# 13. Cross-platform reproducibility requirements

A conformant POP-2-COMPUTE implementation MUST:

## 13.1 Bit-identical output across backends

The same workload MUST produce identical output bytes on:

- x86_64 CPU (Linux, Windows, macOS),
- ARM64 CPU (Linux, macOS),
- NVIDIA CUDA (Linux, Windows),
- AMD ROCm (Linux),
- Apple Metal (macOS).

## 13.2 Bit-identical POP-1 derivation

Given identical output bytes, POP-1 fields MUST be bit-identical
across all platforms.

## 13.3 Bit-identical POP-2-COMPUTE derivation

Given identical POP-1 and identical workload declaration, POP-2
fields MUST be bit-identical.

## 13.4 Test vectors

Each registered kernel MUST publish test vectors covering:

- minimal workload size,
- representative workload size,
- maximum supported workload size,
- known-answer outputs for verification.

---

# 14. Hardware accounting (informative)

POP-2-COMPUTE proof captures cryptographic identity of compute work,
not hardware authenticity.

A device claiming to be a GPU may compute on CPU. This document does
not detect such substitution.

Hardware authenticity is the subject of a future PCIP. Until that
specification is adopted, deployments MAY use:

- timing constraints (compute must complete within window inconsistent
  with CPU-only execution),
- HSM-attested device identity,
- TPM measurements,

as out-of-band signals. These are NOT part of POP-2-COMPUTE canonical
proof.

---

# 15. Validation requirements

A conformant POP-2-COMPUTE validator MUST perform:

## 15.1 Structural validation

Per PCIP-0002 §11.1, plus:

- valid `ComputeWorkloadV1` declaration,
- known `kernel_id`,
- supported `kernel_version`,
- non-zero `n_iterations`,
- non-zero `output_size_bytes`.

## 15.2 Workload validation

Validator MUST:

- reconstruct canonical compute task,
- execute on conformant backend,
- obtain output bytes,
- assert output bytes match POP-1 binding hashes.

## 15.3 POP-2 recomputation validation

Per PCIP-0002 §11.2.

If any recomputed field differs, POP-2-COMPUTE proof MUST be rejected.

# 16. Hash type requirements

`Hash32` per PCIP-0002 §15.

# 17. Canonical encoding requirements

Per PCIP-0002 §16. POP-2-COMPUTE inherits POP-2 canonical encoding
rules. The `ComputeWorkloadV1` declaration is encoded as a separate

# 18. Versioning

POP-2-COMPUTE profile version 1.

```text
profile_version = 1
```

Profile version is independent of POP-2 schema version.

`ComputeWorkloadV1` schema version 1.

# 19. Error handling

Validators MUST reject POP-2-COMPUTE objects under:

- unknown `kernel_id`,
- unsupported `kernel_version`,
- workload reconstruction failure,
- output bytes recomputation mismatch,
- POP-1 binding hash mismatch,
- POP-2 derivation mismatch (per PCIP-0002 §19),
- replay linkage mismatch.

---

# 20. Security considerations

## 20.1 Backend substitution risk

Profile does not detect backend substitution (claimed CUDA, actual CPU).
Deployments MAY use timing constraints as out-of-band signal.

## 20.2 Workload precomputation risk

If workload is small enough that an attacker can precompute outputs,
proof has no value. Deployments MUST select workload sizes inconsistent
with precomputation budgets.

## 20.3 Determinism erosion risk

Future kernel implementations MUST preserve bit-identical output.
Any change in canonical output MUST trigger `kernel_version` increment.

## 20.4 Replay risk

Per PCIP-0002 §20.3.

## 20.5 Hidden scoring risk

Per PCIP-0002 §20.4. POP-2-COMPUTE scoring profile is published.

# 22. Conformance requirements

## 22.1 POP-2-COMPUTE Writer

A writer MUST:

- accept valid `ComputeWorkloadV1`,
- execute workload on conformant backend,
- bind output bytes per §6,
- derive POP-1 per PCIP-0001,
- derive POP-2 per PCIP-0002,
- apply POP-2-COMPUTE scoring profile per §10,
- apply POP-2-COMPUTE MBW transform per §11.

## 22.2 POP-2-COMPUTE Validator

A validator MUST:

- perform structural validation per §15.1,
- perform workload validation per §15.2,
- perform POP-2 recomputation per §15.3,
- reject mismatched proofs.

## 22.3 POP-2-COMPUTE Reader

A reader MUST:

- parse POP-2 objects with POP-2-COMPUTE profile,
- preserve workload declaration semantics,
- reject malformed declarations.

# 26. Implementation guidance

## 26.1 Kernel selection

Initial kernels SHOULD be cryptographic primitives with bit-identical
output by construction:

- `sha256-chain-v1`: iterative SHA-256 (deterministic, integer-only),
- `blake3-tree-v1`: BLAKE3 tree hashing,
- `chacha20-stream-v1`: ChaCha20 keystream generation.

Floating-point kernels SHOULD be avoided in initial deployment.

## 26.2 Workload size

Initial kernel deployments SHOULD use workload sizes that:

- complete within bounded time on target hardware,
- exceed precomputation feasibility for adversaries,
- generate sufficient output bytes for meaningful POP-1 metrics.

## 26.3 Cross-platform validation

Reference implementation SHOULD demonstrate bit-identical compute on:

- x86_64 Linux CPU,
- x86_64 Linux + NVIDIA CUDA,
- ARM64 Linux CPU,
- x86_64 Windows CPU,
- x86_64 Windows + NVIDIA CUDA.

---

# 27. Final statement

POP-2-COMPUTE is not a new POP layer.

POP-2-COMPUTE is a normative profile of POP-2 for active compute proofs.

The canonical proof object remains POP-2 per PCIP-0002.  
The canonical container remains POP-1 per PCIP-0001.  
The canonical replay chain remains POP-1 `prev_capture_hash` semantics.

POP-2-COMPUTE adds:

- workload declaration object (`ComputeWorkloadV1`),
- compute backend abstraction,
- compute scoring profile,
- compute MBW transform,
- cross-platform reproducibility requirements,
- workload reconstruction validation.

POP-2-COMPUTE preserves:

- POP-1 / POP-2 semantics,
- ledger neutrality,
- transport neutrality,
- validator independence,
- recomputability.
