# PCIP-0011
## Validator Tier Registry — Hardware-Measured Capability Tiers

**Document ID:** PCIP-0011  
**Title:** Validator Tier Registry  
**Version:** 0.1-draft  
**Status:** Draft  
**Language:** English  
**Intended category:** Normative Specification  
**Depends on:** PCIP-0001 (POP-1), PCIP-0002 (POP-2), PCIP-0010 (POP-2-COMPUTE)  
**Date:** 2026-05-10

---

# 1. Scope

This document defines **Validator Tier Registry**, a normative on-chain
mechanism for binding validator identity to deterministically measurable
hardware capability classes.

PCIP-0011 specifies:

- canonical tier taxonomy,
- canonical capability declarations,
- hardware-measured tier admission proofs,
- tier-dependent MBW multipliers,
- proof type eligibility by tier,
- tier registration and revocation rules,
- canonical tier proof object.

PCIP-0011 does **not** define:

- specific hardware vendor selection,
- KYC or legal validator identity,
- transport mechanisms for tier proofs,
- consensus rules beyond proof admission gating,
- tokenization conversion rates beyond deterministic multipliers.

This document is **normative**. Tier multipliers and admission rules
defined here are part of the canonical proof admission policy.

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**,
**SHALL NOT**, **SHOULD**, **SHOULD NOT**, **MAY** are to be

# 3. Purpose

PCIP-0011 exists to bind validator economic status to verifiable
hardware capability, replacing self-declaration with mechanical proof.

A valid tier registration enables the following:

1. tier-dependent MBW emission scaling,
2. proof type admission gating per tier,
3. capex-aware reward distribution,
4. deterministic validator class membership on-chain,
5. validator-independent tier verification.

Tier membership is not a label.  
Tier membership is the **outcome of a verifiable hardware proof**.

---

# 4. Design principles

PCIP-0011 SHALL follow the principles below.

## 4.1 Hardware-measured admission

Tier admission MUST be the deterministic result of a hardware capability
proof. No self-declaration alone is admissible.

## 4.2 Mechanical verification

Every tier admission MUST be verifiable by any conformant validator
without trust in the registering party.

## 4.3 Determinism

Tier admission, multipliers, and revocations MUST be deterministic
functions of on-chain state.

## 4.4 Canonicality

The tier registry MUST have exactly one canonical representation per
chain height for each `vid`.

## 4.5 Recomputability

A validator MUST be able to recompute current tier state from chain
history without external lookup.

## 4.6 Capex-proportional emission

Higher hardware tiers MUST receive proportionally higher MBW multipliers
to reflect capital-intensive infrastructure.

## 4.7 Tier neutrality of POP-1 / POP-2

POP-1 and POP-2 canonical proof objects remain tier-neutral. Tier
information is part of admission policy, not canonical proof content.

# 5. Tier taxonomy

The canonical tier set is:

```text
ValidatorTier
 ├─ T1   — CPU baseline
 ├─ T2   — Compute-capable (GPU or equivalent)
 ├─ T3   — Industrial single-node (HSM-anchored)
 ├─ T4   — Industrial multi-node (HSM-cluster cosigned)
 └─ T_AI — AI inference capable
```

Each tier represents a distinct capability class. Tiers are **not**
strictly ordered for all proof types; instead, each proof type
declares which tier set is eligible for admission.

## 5.1 T1 — CPU baseline

**Required capability:** general-purpose CPU.  
**Required proof:** none beyond standard POP-1 / POP-2 derivation.  
**Eligible proof types:** file integrity (POP-2 v1 default profile).  
**MBW multiplier:** 1.0× (canonical baseline).

T1 is the default tier for any validator that does not register a
higher tier.

## 5.2 T2 — Compute-capable

**Required capability:** dedicated compute accelerator (GPU, custom
ASIC, or equivalent) with measurable parallel compute throughput.  
**Required proof:** §7.1 GPU throughput proof.  
**Eligible proof types:** POP-2-COMPUTE (per PCIP-0010).  
**MBW multiplier:** 1.5× (canonical).

T2 admission requires proof that the registering validator can execute
a registered compute kernel at a rate substantially exceeding CPU
baseline.

## 5.3 T3 — Industrial single-node

**Required capability:** hardware security module (HSM) with private
key custody and on-device cryptographic operations.  
**Required proof:** §7.2 HSM attestation proof.  
**Eligible proof types:** all T1, T2 proofs plus HSM-anchored
process evidence.  
**MBW multiplier:** 2.0× (canonical).

T3 admission requires proof that validator signing key resides in
non-exportable HSM custody verifiable through device attestation
challenge.

## 5.4 T4 — Industrial multi-node

**Required capability:** cluster of N >= 2 T3-equivalent nodes producing
co-signed evidence.  
**Required proof:** §7.3 cluster cosignature proof.  
**Eligible proof types:** all T1, T2, T3 proofs.  
**MBW multiplier:** 3.0× (canonical).

T4 admission requires multi-node co-signed POP-2 derivation per
PCIP-0001 §12 replay linkage and cosignature aggregation.

## 5.5 T_AI — AI inference capable

**Required capability:** verifiable AI inference workload execution
with deterministic kernel-anchored output.  
**Required proof:** §7.4 AI inference kernel proof (specification of
the AI kernel registry is deferred to future PCIP).  
**Eligible proof types:** AI inference proofs (future profile).  
**MBW multiplier:** 4.0× (canonical).

T_AI admission requires both hardware capability proof and registered
AI kernel attestation.

# 6. Capabilities

A capability is a normative declaration that a validator possesses a
specific verifiable hardware property.

The canonical capability set is:

```text
Capability
 ├─ CpuBaseline                 (always present)
 ├─ ComputeGpuCuda
 ├─ ComputeGpuRocm
 ├─ ComputeGpuMetal
 ├─ ComputeAsicCustom(string)
 ├─ HsmTpm20
 ├─ HsmZymkey
 ├─ HsmSecureElement(string)
 ├─ ClusterCosignXmssMt
 └─ AiInferenceKernel(string)
```

Capabilities are declared in tier registration proofs (§7).

Each declared capability MUST be backed by an admission proof.

Capabilities MAY be revoked per §10.

# 7. Tier admission proofs

## 7.1 T2 GPU throughput proof

A conformant T2 admission proof MUST consist of:

1. A POP-2-COMPUTE proof per PCIP-0010 using kernel
   `sha256-chain-v1-bench` (or another registered benchmark kernel),
2. measured throughput exceeding tier admission threshold:
   ```text
   measured_hashes_per_second > T2_admission_threshold_hps
   ```
3. canonical `kernel_id`, `kernel_version`, `n_iterations` recorded
   in proof object,
4. timing constraint binding:
   ```text
   capture_end_unix - capture_start_unix < T2_max_duration_seconds
   ```
   such that achieving the measured throughput on CPU is infeasible
   within the declared window.

Canonical constants:

```text
T2_admission_threshold_hps          = 100_000_000  (100 MH/s)
T2_max_duration_seconds             = 60
T2_min_iterations                   = 6_000_000_000  (6 billion)
```

A POP-2-COMPUTE proof satisfying §7.1 admits validator to T2.

Validators MUST recompute the throughput claim using:

```text
hashes_per_second = n_iterations / (capture_end_unix - capture_start_unix)
```

If recomputed value falls below threshold, admission MUST be rejected.

## 7.2 T3 HSM attestation proof

A conformant T3 admission proof MUST consist of:

1. T2 admission proof (T3 implies T2 capability set or higher),
2. HSM challenge response binding:
   - chain-issued challenge nonce (32 bytes from canonical entropy
     source at registration height),
   - HSM-signed response over `(challenge_nonce || vid || pubkey)`,
   - vendor attestation certificate chain rooted in a vendor CA
     registered in a future companion PCIP.

If HSM cannot produce non-exportable signing proof, T3 admission MUST
be rejected.

Acceptable HSM categories (initial set):

- TPM 2.0 conformant devices,
- Zymkey HSM6 (and successors),
- vendor-specific secure elements with documented attestation flow.

Specific vendor CA registry is deferred to future companion PCIP.
Until that document is adopted, T3 admission MAY be implemented as
provisional and re-verified once vendor CA registry is canonical.

## 7.3 T4 cluster cosignature proof

A conformant T4 admission proof MUST consist of:

1. T3 admission proof for each cluster member,
2. cluster cosignature evidence:
   - N >= 2 distinct `vid` values, all at T3 or higher,
   - co-signed POP-2 derivation over shared `container_hash`,
   - canonical aggregation rules defined by cosignature protocol
     specification (out of scope for this document; reference future
     PCIP for cosignature semantics; XMSS^MT subtree cosignature
     anticipated as canonical mechanism).

T4 admission MUST be rejected if any cluster member fails T3.

T4 admission MUST be rejected if cosignature aggregation does not
match canonical aggregation rules.

## 7.4 T_AI inference kernel proof

A conformant T_AI admission proof MUST consist of:

1. T2 admission proof (T_AI implies compute capability),
2. AI inference kernel proof:
   - registered AI kernel identifier,
   - deterministic AI inference output bytes,
   - bit-identical output across permitted backends,
   - canonical kernel registry entry (future PCIP).

AI inference kernel registry, output determinism rules, and admission
thresholds are deferred to future companion PCIP.

# 8. Tier proof object

The canonical tier admission proof object is:

```text
TierAdmissionProofV1 {
  version: u32,
  vid: u32,
  pubkey: PublicKey,
  declared_tier: ValidatorTier,
  declared_capabilities: Vec<Capability>,
  admission_pop2_proof_hashes: Vec<Hash32>,
  challenge_nonce: Hash32,
  attestation_signatures: Vec<AttestationSignature>,
  admission_height: u64,
  proof_hash: Hash32
}
```

## 8.1 Fields

### version
Type: u32. REQUIRED. MUST equal 1 for this specification.

### vid
Type: u32. REQUIRED. Validator identifier within the chain.

### pubkey
Type: PublicKey. REQUIRED. Canonical validator signing public key.

### declared_tier
Type: ValidatorTier (enum). REQUIRED. Target tier for this admission.

### declared_capabilities
Type: Vec<Capability>. REQUIRED. MUST list all capabilities required
for the declared tier. MUST NOT list capabilities not backed by
admission proofs.

### admission_pop2_proof_hashes
Type: Vec<Hash32>. REQUIRED. References to canonical POP-2 proof
hashes that serve as admission evidence. MUST be non-empty for any
tier above T1.

### challenge_nonce
Type: Hash32. REQUIRED. Chain-issued nonce at registration height.
For T1 admissions MAY equal zero hash.

### attestation_signatures
Type: Vec<AttestationSignature>. REQUIRED for T3+. MUST contain
HSM-signed responses per §7.2 / §7.3.

### admission_height
Type: u64. REQUIRED. Chain height at which admission proof is
submitted.

### proof_hash
Type: Hash32. REQUIRED. Canonical hash of all preceding fields.

# 9. MBW multipliers

The canonical tier multiplier table is:

```text
T1   : 1.0×  (mbw_multiplier_q32 = 0x1_0000_0000)
T2   : 1.5×  (mbw_multiplier_q32 = 0x1_8000_0000)
T3   : 2.0×  (mbw_multiplier_q32 = 0x2_0000_0000)
T4   : 3.0×  (mbw_multiplier_q32 = 0x3_0000_0000)
T_AI : 4.0×  (mbw_multiplier_q32 = 0x4_0000_0000)
```

Multipliers are expressed in Q32 fixed-point form (32-bit fractional
part) to preserve determinism across architectures.

## 9.1 Application

For any POP-2 proof admitted under tier T, the effective MBW is:

```text
mbw_effective_u64 = saturating_mul_q32(mbw_u64, multiplier_q32(T))
```

where `saturating_mul_q32` is defined as:

```text
saturating_mul_q32(x, m_q32) = min(u64::MAX, (x * m_q32) >> 32)
```

## 9.2 Determinism

`mbw_effective_u64` MUST be computed by every validator deterministically.
The canonical `mbw_u64` of the POP-2 object remains unchanged.
Multipliers are admission-policy transformations applied at MBW
accounting time.

## 9.3 Governance

Multipliers in §9 are canonical for protocol version 1. Future
governance proposals MAY adjust multipliers via on-chain governance
mechanisms defined in a future PCIP. Until such mechanism is canonical,
multipliers in §9 are immutable.

# 10. Tier revocation

Tier admission MAY be revoked under any of:

1. HSM attestation expiry,
2. detected hardware substitution (out-of-band evidence per future PCIP),
3. cluster member failure (for T4),
4. cryptographic key compromise,
5. governance-initiated revocation via canonical governance mechanism
   (future PCIP).

Upon revocation:

- validator tier reverts to T1,
- pending POP-2-COMPUTE proofs MUST be re-evaluated against admission
  rules at the lower tier,
- MBW multiplier MUST revert to T1 baseline from revocation height
  forward.

Revocation MUST NOT alter prior accepted POP-2 proofs retroactively.
Past `chain_mbw_v0` accumulator state is immutable.

---

# 11. Proof type eligibility

A normative admission eligibility matrix:

```text
Proof type                      Eligible tiers
─────────────────────────────────────────────
POP-2 file integrity            T1, T2, T3, T4, T_AI
POP-2-COMPUTE                   T2, T3, T4, T_AI
HSM-anchored process evidence   T3, T4
Cluster-cosigned proofs         T4
AI inference proofs (future)    T_AI
```

A proof submitted by a validator whose current tier is not in the
eligible set for that proof type MUST be rejected by admission logic.

# 12. Validation requirements

A conformant tier admission validator MUST perform:

## 12.1 Structural validation

- supported `version`,
- valid `vid`,
- non-zero `pubkey`,
- declared tier in canonical taxonomy,
- declared capabilities matching declared tier requirements,
- non-empty `admission_pop2_proof_hashes` for tiers above T1,
- present `attestation_signatures` for T3+ admissions,
- valid `admission_height`.

## 12.2 Admission evidence validation

- each referenced POP-2 proof hash MUST exist in canonical chain state,
- each referenced POP-2 proof MUST recompute correctly,
- POP-2-COMPUTE proofs MUST meet admission thresholds per §7.1,
- HSM attestation signatures MUST verify per §7.2,
- cluster cosignature aggregation MUST verify per §7.3.

## 12.3 Replay enforcement

- admission MUST be linked to canonical chain height,
- duplicate admission for same `vid` at same height MUST be rejected.

## 12.4 Recomputation validation

A validator MUST be able to recompute current tier state for any `vid`
from chain history without external trust.

# 16. Error handling

Validators MUST reject tier admissions under any of:

- unsupported `version`,
- empty or invalid `vid`,
- empty `pubkey`,
- unknown tier,
- missing required capability declarations,
- missing required admission proofs,
- POP-2 proof reference mismatch,
- HSM attestation verification failure,
- threshold below admission requirement,
- replay linkage mismatch,
- duplicate admission for same `vid` at same height.

# 17. Security considerations

## 17.1 Hardware substitution risk

T2 admission proof relies on timing constraint (§7.1) to bind compute
to dedicated accelerator hardware. An attacker with extremely high
CPU performance MAY in principle satisfy threshold; the admission
threshold MUST be set high enough to render this impractical.

## 17.2 HSM attestation revocation

HSM vendor compromise MAY invalidate T3 admission proofs retroactively.
PCIP-0011 does not retroactively rescore past proofs but MUST allow
forward-looking revocation per §10.

## 17.3 Cluster Sybil risk

T4 cluster cosignature requires N >= 2 distinct `vid` at T3 or higher.
Sybil attacks at HSM level are constrained by hardware attestation cost.

## 17.4 Tier upgrade replay

Tier admission proofs MUST be bound to admission height (§8.1) to
prevent replay of past admission proofs as current admissions.

## 17.5 Multiplier governance

Until canonical governance mechanism is adopted, multipliers in §9
are immutable. Ad-hoc multiplier changes MUST NOT be applied.

# 18. Interoperability

PCIP-0011 is:

- ledger-agnostic in its proof object format,
- transport-agnostic,
- validator-independent in admission verification,
- backwards compatible with POP-1 / POP-2 canonical objects.

PCIP-0011 admission rules apply only to PopChain admission policy.

POP-1 and POP-2 objects produced under any tier remain canonically
valid as POP-1 / POP-2 objects independently of tier admission state.

# 19. Relationship to PCIP-0010

PCIP-0010 (POP-2-COMPUTE) defines the canonical compute proof profile.

PCIP-0011 defines:

- which validators may submit POP-2-COMPUTE proofs (§11),
- how compute throughput evidence admits validators to T2 (§7.1),
- how MBW from POP-2-COMPUTE proofs is multiplied at accounting (§9).

PCIP-0010 §22.1 Writer requirements are unchanged. PCIP-0010 admission
gating is now PCIP-0011 §11 (eligible tier set).

# 21. Conformance requirements

## 21.1 Tier registration Writer

A writer MUST:

- prepare valid `TierAdmissionProofV1`,
- bind admission to canonical chain height,
- include all required admission evidence,
- compute canonical `proof_hash`.

## 21.2 Tier registration Validator

A validator MUST:

- perform structural validation per §12.1,
- verify admission evidence per §12.2,
- enforce replay rules per §12.3,
- update canonical tier registry deterministically.

## 21.3 Tier-aware admission Validator

An admission-layer validator MUST:

- look up current tier for submitting `vid`,
- apply eligibility matrix per §11,
- apply MBW multiplier per §9,
- reject proofs ineligible for current tier.

## 21.4 Tier registry Reader

A reader MUST:

- parse canonical tier registry state,
- expose tier per `vid`,
- expose capabilities per `vid`,
- expose admission heights.

# 24. Implementation guidance

## 24.1 T1 default

Validators without explicit admission registration MUST default to T1.
No registration proof is required for T1.

## 24.2 T2 first deployment

Initial T2 admission threshold (§7.1) is set conservatively to reflect
mid-range consumer GPU baseline. Threshold MAY be raised via governance
once hardware distribution data is available.

## 24.3 T3 first deployment

Initial T3 admission relies on a small vendor CA set (TPM 2.0 + Zymkey).
Vendor CA registry expansion is deferred to future companion PCIP.

## 24.4 T4 first deployment

T4 admission requires cosignature protocol implementation. T4 admission
SHOULD remain provisional until cosignature canonical specification is
adopted.

## 24.5 T_AI first deployment

T_AI admission is deferred pending AI inference kernel profile
specification. T_AI tier exists in taxonomy but is not yet admissible.

## 24.6 Genesis tier state

At genesis, no validator is registered at any tier.  
All validators start at T1 (default).  
Tier admissions are submitted post-genesis via canonical mechanism.

# 26. Final statement

PCIP-0011 binds validator economic status to verifiable hardware
capability.

Tier membership is not declared.  
Tier membership is **proven**.

Higher tiers represent higher capital expenditure and operational
complexity. Their MBW multipliers reflect this asymmetry.

PCIP-0011 preserves:

- POP-1 and POP-2 canonical proof neutrality,
- ledger neutrality,
- transport neutrality,
- validator independence,
- recomputability.

PCIP-0011 adds:

- canonical tier taxonomy,
- hardware-measured admission proofs,
- canonical MBW multipliers,
- proof type eligibility matrix,
- tier registry as part of canonical chain state.

The tier registry is part of the canonical chain.  
Tier admission is part of the canonical proof admission policy.  
Tier multipliers are part of the canonical MBW accounting.
