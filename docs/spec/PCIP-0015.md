# PCIP-0015
## Network Phase Genesis Continuation — Alpha to Beta to Mainnet

**Document ID:** PCIP-0015  
**Title:** Network Phase Genesis Continuation  
**Version:** 0.1-draft  
**Status:** Draft  
**Language:** English  
**Intended category:** Normative Specification  
**Depends on:** PCIP-0001 (POP-1), PCIP-0002 (POP-2), PCIP-0011 (Tier Registry)  
**Date:** 2026-05-11

---

# 1. Scope

This document defines the **canonical mechanism for chain history
continuation** across PopChain network phase transitions:

- alpha testnet to beta testnet,
- beta testnet to mainnet.

PCIP-0015 specifies:

- canonical phase transition model,
- alpha tip snapshot binding,
- beta genesis state composition,
- alpha-to-beta history continuation proof,
- beta-to-mainnet history continuation proof,
- canonical verification by future validators,
- history immutability guarantees from alpha block 0 forward.

PCIP-0015 does **not** define:

- governance decision process for phase transitions,
- token economics or supply migration between phases,
- legal entity transitions (foundation incorporation),
- specific timing or duration of beta phase,
- mainnet launch criteria beyond canonical state continuity.

This document is **normative**. The chain history continuation
mechanism is part of the canonical PopChain identity.

# 3. Purpose

PCIP-0015 exists to guarantee that PopChain chain history is
preserved across network phase transitions. Every block from
alpha-testnet block 0 forward is permanent on-chain history,
inherited by beta and mainnet.

A canonical phase transition model enables:

1. immutable history from alpha-testnet inception forward,
2. verifiable continuity by future validators independent of
   foundation trust assumptions,
3. clear semantics for application developers depending on past
   on-chain events,
4. economic continuity (validator stake credits, balances, MBW
   accumulators carried forward),
5. mechanical proof of chain authenticity at mainnet launch.

PopChain rejects the model where mainnet launches as fresh genesis
with no prior history. Mainnet block 0 MUST contain the full
chain history of alpha and beta phases.

---

# 4. Design principles

PCIP-0015 SHALL follow the principles below.

## 4.1 History immutability

No block from alpha-testnet block 0 forward MAY be retroactively
modified. Chain phase transitions preserve all blocks intact.

## 4.2 State continuity

Validator stake credits, validator minted bincoin accumulators,
machine capture chain state, balances, and AMM reserves MUST be
carried forward across phase transitions without loss.

## 4.3 Cryptographic continuation proof

Each phase transition MUST be accompanied by a cryptographic
continuation proof binding the new phase genesis state to the
prior phase tip.

## 4.4 Validator-independent verification

Any future validator MUST be able to verify chain history continuity
from alpha-testnet block 0 forward without relying on foundation
trust assumptions.

## 4.5 No fork retroactive admission

Phase transitions MUST select the canonical alpha tip via
deterministic fork choice at transition height. Orphan blocks
remain orphaned; only the canonical chain is carried forward.

## 4.6 Single canonical history

There MUST be exactly one canonical chain history per PopChain
identity. Alternative histories produced by minority forks MUST NOT
be admitted as canonical post-transition.

---

# 5. Network phase model

PopChain operates in three canonical network phases:

## 5.1 Alpha phase (current)

**Designation:** `popchain-alpha-testnet`  
**Chain ID:** as declared in alpha genesis.  
**Operational status:** active since alpha block 0.  
**Purpose:** mechanical proof that protocol logic operates correctly.  
**Validator set:** founder-operated and early collaborator validators.  
**Source code:** `popchain-core` private repository during pre-beta.  
**Token economics:** testnet economics, non-binding for mainnet supply.

## 5.2 Beta phase

**Designation:** `popchain-beta-testnet`  
**Chain ID:** distinct from alpha chain ID.  
**Operational status:** post alpha-to-beta transition.  
**Purpose:** validator economy, smart contracts, industrial pilots,
sales of validator hardware, foundation organizational structure.  
**Validator set:** open admission via PCIP-0011 tier admission proofs,
foundation-operated bootstrap nodes, third-party industrial validators.  
**Source code:** beta source available publicly per governance.  
**Token economics:** beta-phase token model, possibly distinct from
mainnet supply.  
**Duration:** indeterminate; ends when mainnet launch criteria are met.

## 5.3 Mainnet phase

**Designation:** `popchain-mainnet`  
**Chain ID:** distinct from alpha and beta chain IDs.  
**Operational status:** post beta-to-mainnet transition.  
**Purpose:** production network operation.  
**Validator set:** fully open, decentralized validator admission per
PCIP-0011, no special foundation role beyond canonical participation.  
**Source code:** publicly available, fully open source.  
**Token economics:** canonical mainnet token model.  
**Duration:** continuous operation from mainnet block 0 forward.

# 6. Phase transition canonical object

The canonical object representing a phase transition is:

```text
PhaseTransitionV1 {
  version: u32,
  from_phase: NetworkPhase,
  to_phase: NetworkPhase,
  from_chain_id: String,
  to_chain_id: String,
  from_tip_block_hash: Hash32,
  from_tip_height: u64,
  from_state_root: Hash32,
  from_chain_mbw_v0: u128,
  transition_unix: u64,
  carry_over_state_blob_hash: Hash32,
  continuation_proof_hash: Hash32,
}
```

## 6.1 Fields

### version
Type: u32. REQUIRED. MUST equal 1 for this specification.

### from_phase
Type: NetworkPhase enum. REQUIRED. MUST be a defined network phase
per §5.

### to_phase
Type: NetworkPhase enum. REQUIRED. MUST be a defined network phase
strictly later than `from_phase`.

Permitted transitions:
- AlphaTestnet -> BetaTestnet
- BetaTestnet -> Mainnet

### from_chain_id
Type: string. REQUIRED. MUST equal the chain ID of the source phase.

### to_chain_id
Type: string. REQUIRED. MUST equal the chain ID of the destination
phase. MUST differ from `from_chain_id`.

### from_tip_block_hash
Type: Hash32. REQUIRED. The canonical tip block hash of the source
phase at the transition height.

### from_tip_height
Type: u64. REQUIRED. The canonical tip height of the source phase.

### from_state_root
Type: Hash32. REQUIRED. The canonical state-root hash at the source
phase tip, computed per PCIP-0011 §12.4 (state-root v4 or higher).

### from_chain_mbw_v0
Type: u128. REQUIRED. The total chain MBW accumulator value at the
source phase tip.

### transition_unix
Type: u64. REQUIRED. Unix timestamp of the canonical transition event.

### carry_over_state_blob_hash
Type: Hash32. REQUIRED. SHA-256 hash of the canonical serialized
source phase final state blob (§7).

### continuation_proof_hash
Type: Hash32. REQUIRED. Canonical proof hash binding all preceding
fields per §8.

---

# 7. Carry-over state blob

The carry-over state blob is the canonical serialized representation
of the source phase final state used as the seed for the destination
phase genesis.

## 7.1 Canonical content

The carry-over state blob MUST contain, at minimum:

- source phase final `StateV1` per PCIP-0011 schema,
- source phase final tip block header,
- source phase final epoch finalization data,
- source phase validator tier registry per PCIP-0011 §12.4,
- source phase machine capture chain heads per machine,
- source phase balance ledger,
- source phase pUSDC ledger,
- source phase AMM reserves,
- source phase validator stake credits,
- source phase validator minted bincoin accumulators,
- canonical metadata describing phase transition.

## 7.2 Canonical encoding

The carry-over state blob MUST be canonically encoded such that:

- field ordering is deterministic,
- map iteration is deterministic (sorted by canonical key order),
- numeric encoding is little-endian fixed-width,
- string encoding is UTF-8 with explicit length prefixes,
- hash values are raw 32-byte sequences.

Canonical encoding rules for the blob are deferred to PCIP-0003
(canonical encoding spec, planned).

## 7.3 Blob hash

`carry_over_state_blob_hash` MUST equal:

```text
SHA-256(canonical_encoded_state_blob)
```

This hash binds the destination genesis to the exact source state
snapshot.

---

# 8. Continuation proof

The continuation proof is the canonical cryptographic binding
between source phase tip and destination genesis.

## 8.1 Canonical derivation

`continuation_proof_hash` MUST be derived as:

```text
SHA-256(
  b"popchain:phase-continuation:v1",
  version,
  from_phase_byte,
  to_phase_byte,
  from_chain_id_bytes,
  to_chain_id_bytes,
  from_tip_block_hash,
  from_tip_height,
  from_state_root,
  from_chain_mbw_v0,
  transition_unix,
  carry_over_state_blob_hash
)
```

Where each field contributes its canonical byte representation per
encoding rules in §7.2.

## 8.2 Verifiability

Any future validator MAY verify `continuation_proof_hash` by:

1. acquiring the canonical carry-over state blob,
2. independently computing `carry_over_state_blob_hash`,
3. independently computing `continuation_proof_hash` per §8.1,
4. comparing to the published continuation proof hash.

Identical hash recomputation confirms continuation proof authenticity.

---

# 9. Destination genesis composition

## 9.1 Destination block 0 header

The destination phase block 0 (genesis block) header MUST contain:

- destination chain ID,
- destination version,
- height = 0,
- `prev_block_hash` = canonical hash of source phase final tip block
  (equal to `from_tip_block_hash` in PhaseTransitionV1),
- `phase_transition_hash` = `continuation_proof_hash` from
  PhaseTransitionV1.

The destination block 0 MUST reference the source phase tip directly.

## 9.2 Destination block 0 state

The destination phase block 0 state MUST initialize from the
canonical carry-over state blob:

- copy `chain_mbw_v0` from source state (preserving accumulator),
- copy validator stake credits per validator,
- copy validator minted bincoin accumulators per validator,
- copy balances per address,
- copy pUSDC balances and locked balances,
- copy AMM reserves,
- copy validator tier registry per PCIP-0011,
- copy machine capture chain heads per machine,
- reset destination phase-specific fields as specified per phase
  transition rules.

## 9.3 Destination block 0 state root

The destination block 0 state root MUST be computed per PCIP-0011
§12.4 state-root v4 (or higher version applicable at destination
phase launch) over the initialized destination state.

## 9.4 Recomputation guarantee

Any conformant validator running the canonical Rust implementation
at the destination phase launch height MUST be able to:

1. fetch the canonical carry-over state blob,
2. construct destination block 0 state per §9.2,
3. compute destination block 0 state root per §9.3,
4. produce identical state root to that published in destination
   block 0 header.

This recomputation guarantee enables validator-independent verification
of phase transition correctness.

# 10. Alpha-to-Beta transition

## 10.1 Trigger conditions

Alpha-to-beta transition MAY proceed when:

- foundation governance approves transition (legal entity in place,
  beta source code public, beta validator admission process defined),
- alpha source code public availability is established,
- beta network infrastructure is provisioned,
- PCIP-0011 tier admission process is operational,
- canonical carry-over state blob has been published.

## 10.2 Transition execution

1. Select canonical alpha tip block at transition height.
2. Compute canonical state at alpha tip per state-root v4.
3. Construct canonical carry-over state blob per §7.
4. Publish blob to canonical distribution channels.
5. Compute `continuation_proof_hash` per §8.
6. Construct PhaseTransitionV1 object per §6.
7. Launch beta network with block 0 referencing the PhaseTransitionV1
   per §9.

## 10.3 Alpha continuation post-beta-launch

After beta launch, the alpha testnet MAY continue to operate for
historical reference and educational purposes. Alpha post-beta-launch
operation is non-authoritative; the canonical PopChain history
proceeds via beta.

## 10.4 Application impact

Applications and validators built against alpha testnet MUST migrate
to beta. Alpha-testnet RPC endpoints SHOULD continue accepting
read-only queries against historical state but MUST NOT be relied
upon for canonical operation post beta launch.

---

# 11. Beta-to-Mainnet transition

## 11.1 Trigger conditions

Beta-to-mainnet transition MAY proceed when:

- beta operational stability is demonstrated over a multi-month
  period,
- validator decentralization criteria are met (geographic, organi-
  zational, hardware tier distribution),
- mainnet source code is publicly available,
- mainnet governance mechanisms are operational,
- foundation organizational structure is in place,
- bridge infrastructure (PopChain <-> Solana pUSDC) is operational,
- canonical carry-over state blob (alpha + beta history) is
  published.

## 11.2 Transition execution

1. Select canonical beta tip block at transition height.
2. Compute canonical state at beta tip per applicable state-root
   version.
3. Construct canonical carry-over state blob containing full alpha +
   beta history references per §7.
4. Publish blob to canonical distribution channels.
5. Compute `continuation_proof_hash` per §8.
6. Construct PhaseTransitionV1 object per §6.
7. Launch mainnet network with block 0 referencing the
   PhaseTransitionV1 per §9.

## 11.3 Beta continuation post-mainnet-launch

After mainnet launch, the beta testnet MAY continue for historical
reference. Beta post-mainnet-launch operation is non-authoritative.

## 11.4 History completeness in mainnet

Mainnet canonical history MUST reference alpha + beta history via:

- direct reference in PhaseTransitionV1 from beta tip,
- indirect reference through beta's PhaseTransitionV1 to alpha tip,
- full chain history downloadable via canonical distribution from
  alpha block 0 forward.

This means: at mainnet block 0, the canonical history of PopChain
extends back to alpha-testnet block 0, with cryptographic binding
through both PhaseTransitionV1 records.

# 12. Canonical history downloading

A new validator joining at any phase MUST be able to download
canonical chain history from phase block 0 forward.

## 12.1 Distribution channels

Canonical chain history SHOULD be distributable via:

- official PopChain Foundation servers,
- IPFS or equivalent content-addressed distribution,
- mirror nodes operated by community,
- snapshot archives published alongside continuation proofs.

## 12.2 Historical phase blocks

A new mainnet validator MAY:

- download mainnet history starting from mainnet block 0,
- additionally download beta history starting from beta block 0,
- additionally download alpha history starting from alpha block 0,
- verify continuity proofs at phase transition boundaries.

Full validator verification SHOULD include continuity proof
verification at phase boundaries.

## 12.3 Pruned operation

A mainnet validator MAY operate without complete historical block
storage provided:

- continuation proof hashes are verified,
- canonical state at mainnet block 0 is verified,
- mainnet block history is complete from mainnet block 0 forward.

# 13. Validation requirements

A conformant validator MUST verify phase transition by:

## 13.1 Structural validation

- supported PhaseTransitionV1 version,
- defined source and destination phases,
- non-empty chain IDs,
- non-zero source tip block hash,
- non-zero source state root,
- valid transition timestamp,
- non-zero carry-over state blob hash,
- non-zero continuation proof hash.

## 13.2 Continuity validation

- destination block 0 `prev_block_hash` equals `from_tip_block_hash`,
- destination block 0 `phase_transition_hash` equals
  `continuation_proof_hash`,
- `continuation_proof_hash` recomputes correctly per §8.1.

## 13.3 State recomputation

- carry-over state blob hash recomputes correctly,
- destination block 0 state matches initialization from blob,
- destination block 0 state root recomputes correctly per PCIP-0011
  §12.4.

If any validation fails, the destination phase MUST be rejected
as not-canonical-PopChain-continuation.

# 15. Error handling

Conformant validators MUST reject phase transitions under:

- unsupported PhaseTransitionV1 version,
- undefined source or destination phase,
- empty chain ID,
- mismatched `prev_block_hash` in destination block 0,
- mismatched `continuation_proof_hash` recomputation,
- mismatched carry-over blob hash recomputation,
- destination state root recomputation mismatch,
- transition violating permitted phase ordering (e.g., mainnet to
  beta would be rejected).

# 16. Security considerations

## 16.1 Foundation trust during transition

Phase transitions are foundation-initiated events during pre-mainnet
phases. Trust during transition window is mitigated by:

- public canonical carry-over state blob,
- cryptographic continuation proof,
- recomputability by any validator,
- post-transition state verifiability without foundation trust.

After mainnet launch, no further foundation-initiated transitions
are permitted within protocol version 1. Future phase transitions
beyond mainnet would require explicit protocol version bump.

## 16.2 Carry-over state authenticity

The canonical carry-over state blob MUST be publicly available
prior to or simultaneously with destination genesis block 0
announcement. If blob is unavailable, validators MUST NOT initialize
destination state.

## 16.3 Replay protection

Each phase transition is bound to a specific source tip block hash
and source state root. A continuation proof cannot be replayed to
admit a different destination genesis.

## 16.4 Alternative chain admission

Alternative chain histories (forks of alpha that did not become
canonical, alternative beta implementations, dissenting mainnet
launches) MUST NOT be admitted as canonical post-transition. Each
PopChain phase has exactly one canonical history.

## 16.5 Cryptographic continuity strength

Continuation proof strength is SHA-256 over canonically encoded
state blob and PhaseTransitionV1 fields. This provides 128-bit
collision resistance under standard SHA-256 assumptions.

# 17. Interoperability

## 17.1 With PCIP-0011 (Tier Registry)

Validator tier registry per PCIP-0011 is part of carry-over state.
Validators admitted at higher tiers on alpha MAY carry tier status
into beta provided their admission proofs remain valid in beta
canonical infrastructure.

Beta MAY re-admit validators with updated admission proof requirements.

## 17.2 With PCIP-0010 (POP-2-COMPUTE)

Compute proofs submitted on alpha remain part of carry-over history
into beta. Beta validators MAY reference alpha compute proofs as
historical evidence.

## 17.3 With PCIP-0001 (POP-1), PCIP-0002 (POP-2)

POP-1 and POP-2 canonical objects produced under alpha remain
canonical in beta and mainnet. Phase transition does not invalidate
prior proofs.

# 18. Conformance requirements

## 18.1 Phase transition Writer

A writer (foundation, transition coordinator) MUST:

- compute canonical carry-over state blob per §7,
- publish blob to canonical distribution channels,
- compute continuation proof per §8,
- construct PhaseTransitionV1 per §6,
- announce destination genesis with required references per §9.

## 18.2 Phase transition Validator

A validator MUST:

- verify structural validity per §13.1,
- verify continuity per §13.2,
- verify state recomputation per §13.3,
- reject malformed or non-continuous transitions.

## 18.3 Phase transition Reader

A reader (block explorer, third-party tool) MUST:

- parse PhaseTransitionV1 objects per §6,
- expose continuation proof and source tip references,
- preserve semantic meaning of phase identifiers.

# 22. Final statement

PopChain chain history is **permanent** from alpha-testnet block 0
forward.

Network phase transitions (alpha to beta, beta to mainnet) preserve
all prior blocks, state, and accumulators. Each transition is bound
to its source phase by cryptographic continuation proof.

There is **one canonical PopChain history**. Mainnet launch is not
a fresh start. Mainnet block 0 stands on the cumulative cryptographic
foundation of alpha and beta phases.

PCIP-0015 preserves:

- POP-1 and POP-2 canonical proof objects across phases,
- POP-2-COMPUTE compute proof history across phases,
- PCIP-0011 validator tier registry across phases,
- ledger neutrality at proof layer,
- transport neutrality at network layer,
- canonical Rust implementation discipline per PCIP-0020.

PCIP-0015 adds:

- canonical phase model (alpha, beta, mainnet),
- PhaseTransitionV1 canonical object,
- carry-over state blob mechanism,
- continuation proof cryptographic binding,
- validator-independent transition verification,
- single canonical history guarantee.

The alpha block 0 is the foundation.  
Beta block 0 stands on alpha.  
Mainnet block 0 stands on alpha and beta.  
Together they form one continuous canonical PopChain history.
