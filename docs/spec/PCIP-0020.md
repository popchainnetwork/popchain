# PCIP-0020
## Language Discipline — Canonical Implementation Language

**Document ID:** PCIP-0020  
**Title:** Language Discipline  
**Version:** 0.1-draft  
**Status:** Draft  
**Language:** English  
**Intended category:** Normative Specification  
**Depends on:** PCIP-0001 (POP-1), PCIP-0002 (POP-2)  
**Date:** 2026-05-11

---

# 1. Scope

This document defines the **canonical implementation language** for
PopChain protocol logic.

PCIP-0020 specifies:

- which programming language is canonical for PopChain core logic,
- conformance requirements for canonical implementations,
- treatment of third-party reimplementations in other languages,
- cross-platform binary distribution requirements,
- canonical build toolchain expectations,
- determinism guarantees across architectures and operating systems.

PCIP-0020 does **not** define:

- specific Rust edition versions beyond minimum requirements,
- compiler-internal optimizations or inlining decisions,
- third-party library audit policy,
- developer tooling preferences (IDE, editor, linter beyond clippy),
- networking transport-layer implementation language.

This document is **normative**. The choice of canonical implementation
language is part of the canonical protocol identity.

---

# 2. Conformance language

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**,
**SHALL NOT**, **SHOULD**, **SHOULD NOT**, **MAY** are to be
interpreted as normative requirements (RFC 2119 / RFC 8174).

---

# 3. Purpose

PCIP-0020 exists to lock canonical implementation discipline before
mainnet, eliminating ambiguity about which implementation is
authoritative when validators disagree.

A canonical language enables:

1. unambiguous protocol behavior across all conformant validators,
2. single source of truth for protocol logic,
3. predictable determinism across all supported platforms,
4. simplified audit surface for security review,
5. clear expectations for third-party tooling.

Multi-language canonical implementations introduce divergence risk.
Cross-language semantic differences (numeric overflow handling,
floating-point determinism, garbage collection timing, default
hash algorithms, integer width) can cause validators to disagree
on otherwise valid inputs. PCIP-0020 forecloses this risk.

---

# 4. Canonical implementation language

The canonical implementation language for PopChain protocol logic
is **Rust**.

## 4.1 Canonical Rust edition

Conformant implementations MUST target Rust edition 2021 or later.
The reference implementation pins toolchain via `rust-toolchain.toml`
at repository root.

## 4.2 Minimum supported Rust version

The minimum supported Rust version (MSRV) is the latest stable Rust
release at the time of protocol version specification.

For protocol version 1, MSRV is Rust 1.80 or later. MSRV updates
MAY occur via PCIP amendment.

## 4.3 Unsafe code

Protocol-critical modules MUST use `#![forbid(unsafe_code)]` at module
scope. The reference implementation currently applies this attribute
to `src/metrics.rs`, `src/chain.rs`, and other canonical modules.

Unsafe code blocks MAY exist in:

- third-party crate dependencies (subject to audit),
- platform-specific FFI for backend integration (CUDA, ROCm, Metal),
- low-level performance-critical paths with explicit safety
  documentation.

Use of unsafe code in canonical protocol logic MUST be justified
and reviewed.

## 4.4 Toolchain pinning

Conformant implementations SHOULD pin Rust toolchain via
`rust-toolchain.toml`. The reference implementation pins to stable
channel with `rustfmt` and `clippy` components.

---

# 5. Third-party implementations

## 5.1 Definition

A **third-party implementation** is any PopChain protocol
implementation written in a language other than Rust.

## 5.2 Authoritative status

Third-party implementations are **not authoritative**. The canonical
Rust implementation is authoritative for protocol behavior. If a
third-party implementation produces different results than the
canonical implementation for the same input, the canonical
implementation defines correct behavior.

## 5.3 Permitted use

Third-party implementations MAY be developed for:

- educational purposes,
- partial verification tooling (read-only block inspection),
- explorer applications,
- documentation generation,
- testing frameworks,
- bridge software to other ecosystems.

Third-party implementations MUST NOT be used as authoritative
validators in production network operation.

## 5.4 Disclosure requirements

Any third-party implementation distributed for community use SHOULD:

- clearly state non-authoritative status,
- document which canonical Rust version it targets,
- provide test vectors demonstrating conformance with canonical
  implementation for declared scope,
- note divergence cases discovered during development.

## 5.5 PopChain governance disclaimer

Distribution or maintenance of third-party implementations is not
endorsed, sponsored, or warranted by PopChain protocol governance.
Risk of using third-party implementations rests with the user.

# 6. Cross-platform requirements

The canonical Rust implementation MUST produce bit-identical results
for the same input across all supported platforms.

## 6.1 Supported architectures

Tier 1 supported architectures (mandatory cross-platform parity):

- `x86_64-unknown-linux-gnu`
- `aarch64-unknown-linux-gnu`
- `x86_64-pc-windows-msvc`
- `x86_64-apple-darwin`
- `aarch64-apple-darwin`

Tier 2 supported architectures (best-effort, SHOULD work):

- `x86_64-unknown-freebsd`
- `aarch64-unknown-linux-musl`

Tier 3 architectures (experimental, MAY work):

- `wasm32-unknown-unknown` (for browser-side verifier tooling)
- `riscv64gc-unknown-linux-gnu`

## 6.2 Bit-identical output requirement

For canonical protocol operations (POP-1 derivation, POP-2 derivation,
state-root computation, signature verification, deterministic
proof construction), every Tier 1 platform MUST produce identical
output bytes for identical input.

## 6.3 Cross-platform verification

The reference implementation SHOULD ship automated cross-platform
verification via continuous integration:

- matrix build across all Tier 1 architectures,
- test suite execution on each architecture,
- comparison of test output hashes across architectures,
- CI gate that rejects merges producing platform-divergent results.

## 6.4 Floating-point determinism

Protocol-critical code paths MUST NOT use floating-point arithmetic
where bit-identical results across architectures cannot be guaranteed.

Integer arithmetic in fixed-point representation (e.g., Q32 fixed-point
multipliers per PCIP-0011 §9) is preferred for protocol-relevant
mathematical operations.

Floating-point usage in non-protocol code paths (logging output
formatting, telemetry display, user-facing rendering) is permitted.

# 7. Compute backend abstraction

PCIP-0010 defines compute backends (CPU, CUDA, ROCm, Metal, Custom).
PCIP-0020 specifies their canonical implementation:

## 7.1 Backend implementation language

All compute backend implementations MUST be written in Rust.

Native libraries underlying backends (CUDA Toolkit, ROCm libraries,
Metal framework) MAY be in their respective vendor languages (C, C++,
Objective-C). Such native libraries are integrated via Rust foreign
function interface (FFI) bindings.

## 7.2 Backend bindings

Recommended Rust binding crates:

- CUDA: `cudarc` or `cust` crate ecosystem,
- ROCm: appropriate Rust bindings as ecosystem matures,
- Metal: `metal-rs` crate.

Each backend MUST produce output bytes bit-identical to canonical CPU
reference implementation for the same workload declaration.

## 7.3 Backend independence guarantee

A POP-2-COMPUTE proof MUST be verifiable regardless of which backend
produced it. Validator implementations MAY use any conformant backend
for verification.

# 8. Build toolchain requirements

## 8.1 Canonical build tool

The canonical build tool is `cargo` shipped with Rust toolchain.

## 8.2 Reproducible builds

Conformant implementations SHOULD support reproducible builds:

- pinned `Cargo.lock` in repository,
- explicit dependency versions,
- documented build environment (Rust toolchain version, target triple,
  CPU features),
- deterministic compiler flags.

## 8.3 Release profile

Production validators SHOULD build with `--release` profile:

```text
cargo build --release --lib
cargo test --release --lib
```

Debug builds are for development only and SHOULD NOT be deployed
to production validators.

## 8.4 Binary distribution

Pre-built binaries for Tier 1 platforms SHOULD be distributed via:

- official PopChain repository releases (GitHub Releases),
- cryptographic signature of release artifacts,
- SHA-256 checksums published alongside binaries,
- reproducibility documentation.

---

# 9. Determinism requirements

## 9.1 Output determinism

For input I and canonical Rust implementation R producing output O:

```text
R(I)_x86_64_linux   = R(I)_x86_64_windows  = R(I)_aarch64_linux
                    = R(I)_x86_64_macos    = R(I)_aarch64_macos
```

All Tier 1 platforms MUST produce identical O for identical I.

## 9.2 Test vector reproducibility

Test vectors used in canonical KAT tests (PCIP-0011 anchor tests,
POP-1 conformance tests, POP-2 conformance tests) MUST produce
identical results across all Tier 1 platforms.

CI infrastructure MUST verify this across the matrix of Tier 1
platforms on every commit.

## 9.3 Anchor guard tests

Canonical anchor values (multiplier tables, eligibility matrices,
state-root version markers, hash constants) MUST be protected by
anchor guard tests. Any modification of canonical values requires
explicit version bump in the affected PCIP and explicit anchor
test update.

# 10. Reference implementation

## 10.1 Reference implementation location

The canonical reference implementation is hosted at:

```text
github.com/popchainnetwork/popchain-core (private during pre-beta)
github.com/popchainnetwork/popchain      (public mirror for specs
                                          and reproducibility artifacts)
```

## 10.2 Reference implementation status

The reference implementation defines canonical behavior for the
protocol. Any third-party implementation MUST track the reference
implementation for correctness.

## 10.3 Pre-beta to mainnet transition

During pre-beta development, the reference implementation source code
remains in `popchain-core` (private). Specifications, reproducibility
artifacts, and design documents are mirrored to `popchain` (public).

Upon beta launch, the reference implementation source code MAY be
moved to a public repository as governance permits. Mainnet launch
MUST be accompanied by full public source availability.

# 11. Audit and security review

## 11.1 Single-language audit scope

By restricting canonical implementation to Rust, audit scope is
minimized to:

- one programming language,
- one canonical compiler,
- one canonical test framework,
- one canonical dependency manager,
- one canonical build tool.

This minimization is a deliberate security property.

## 11.2 Dependency audit

Canonical implementation MUST limit dependencies to:

- well-established, widely-audited crates,
- minimal dependency tree,
- explicit version pinning,
- documented audit status for security-critical crates.

Security-critical crates (cryptographic primitives, hashing, signing)
SHOULD be from the RustCrypto ecosystem or equivalent peer-reviewed
sources.

## 11.3 Unsafe code audit

All unsafe code blocks in canonical implementation MUST be:

- documented with safety justification,
- reviewed during code review,
- minimized to absolute necessity.

# 12. Versioning and amendments

PCIP-0020 protocol version 1.

```text
protocol_version = 1
```

Amendments to canonical language requirements (MSRV bump, Tier
platform changes, edition updates) MUST proceed via PCIP amendment
process.

Adding additional canonical implementation languages is NOT
permitted under protocol version 1. Such a change would require
PCIP-0020 v2 and explicit protocol version bump.

---

# 13. Error handling

Conformant implementations MUST reject:

- attempts to register non-Rust implementations as authoritative
  validators in canonical operation,
- proof submissions from non-canonical implementations claiming
  authoritative status,
- canonical state transitions produced by non-canonical
  implementations.

Conformant implementations MAY interoperate with non-canonical
implementations for non-authoritative purposes (block inspection,
explorer, education).

---

# 14. Security considerations

## 14.1 Reimplementation drift risk

Permitting multiple authoritative language implementations creates
risk of subtle behavioral divergence (numeric overflow handling,
floating-point precision, default hashing, garbage collection
timing). PCIP-0020 forecloses this risk by selecting a single
canonical language.

## 14.2 Compiler determinism

Rust toolchain (rustc + LLVM) is deterministic for canonical protocol
operations subject to the constraints in §6.4 and §9. Reliance on
this determinism is a known protocol assumption.

If future Rust toolchain releases introduce non-deterministic behavior
in protocol-critical code paths, the reference implementation MUST
adapt to preserve canonical determinism.

## 14.3 Supply chain risk

Single-language canonical implementation concentrates supply chain
risk on Rust toolchain and `crates.io` ecosystem. Mitigations:

- `Cargo.lock` pinning,
- vendored dependencies for critical builds,
- offline build capability,
- mirror availability planning.

## 14.4 Audit cost

Restricting canonical implementation to Rust reduces audit cost
compared to multi-language implementations but concentrates audit
attention on Rust-specific failure modes.

# 15. Interoperability

## 15.1 Protocol-level interoperability

PopChain proofs (POP-1 containers, POP-2 proof objects, POP-2-COMPUTE
proofs) are language-agnostic in their serialized form. A third-party
reader written in any language MAY parse and verify proofs against
canonical encoding rules (PCIP-0003 future).

## 15.2 Validator-level interoperability

Authoritative validators MUST run canonical Rust implementation.
Validator-to-validator network protocol is language-neutral over
the wire, but validator state machine execution MUST be canonical
Rust.

## 15.3 Application-level interoperability

Smart contracts, application logic, and chain-aware applications
MAY be written in other languages provided they:

- interact via canonical RPC interface,
- do not claim authoritative validator status,
- do not influence canonical chain state transitions independently
  of canonical validator logic.

# 16. Migration considerations

## 16.1 Existing code in other languages

PopChain repository history contains Python scripts (`demo_pop6_sim.py`,
`bridge_operator.py`) and JavaScript code (bWalletX wallet
application). These are non-authoritative tooling and remain so.

## 16.2 Educational reimplementations

Educational or reference reimplementations in other languages
(e.g., Python POP-1 parser for documentation purposes) are permitted
under §5.3 with disclosure requirements per §5.4.

## 16.3 No migration to multi-language canonical

Migration of canonical implementation to multi-language status is
NOT permitted under protocol version 1. Any future change to
multi-language canonical status would constitute a fundamental
protocol change requiring full community governance approval.

---

# 17. Conformance requirements

## 17.1 Canonical implementation Writer

An implementation claiming canonical status MUST:

- be written in Rust meeting MSRV per §4.2,
- target Tier 1 architectures per §6.1,
- produce bit-identical output across all Tier 1 architectures,
- pass canonical KAT test suite,
- support release-profile builds per §8.3,
- forbid unsafe code in protocol-critical modules per §4.3.

## 17.2 Canonical implementation Validator

A validator running canonical implementation MUST:

- be the canonical Rust binary or built from canonical source,
- not interoperate with non-canonical authoritative validators,
- reject state transitions produced by non-canonical implementations
  claiming authority.

## 17.3 Third-party implementation Reader

A third-party reader claiming PCIP-0020 conformance MUST:

- clearly state non-authoritative status per §5.4,
- document target canonical Rust version,
- pass canonical conformance test vectors for declared scope,
- not impersonate canonical validator status.

# 18. Test expectations

A conformant test suite SHOULD include:

- compilation success on all Tier 1 architectures,

- floating-point determinism verification where applicable,
- anchor guard test execution on every commit,
- CI matrix validation for cross-platform parity.

# 21. Final statement

The canonical implementation language for PopChain is **Rust**.

Multi-language canonical implementations are not permitted under
protocol version 1.

Third-party implementations in other languages MAY exist for
non-authoritative purposes (education, tooling, explorers) subject
to disclosure requirements.

The canonical Rust implementation defines protocol behavior. When
implementations disagree, the canonical Rust implementation is
authoritative.

PCIP-0020 preserves:

- POP-1 and POP-2 canonical object neutrality (proofs themselves
  remain language-agnostic in serialized form),
- ledger neutrality at the proof layer,
- transport neutrality at the network layer,
- validator independence within the canonical implementation.

PCIP-0020 adds:

- canonical implementation language (Rust),
- canonical toolchain (cargo + rustc + stable channel),
- cross-platform bit-identical output requirement,
- supported architecture tier list,
- third-party reimplementation disclosure rules.

The canonical implementation is one language.  
The canonical platform support is many architectures.  
The canonical output is bit-identical across all of them.
